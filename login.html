      Guest entry — how it works
      
    

    

      
Guest entry lets you in without an account and without storing anything that
      could identify you — no email, no name, no tracking.

      
To keep bots out without a CAPTCHA, your browser quietly solves a small
      proof-of-work puzzle first:

      
    
        
  1. The server hands your browser a one-time challenge tied to your IP.
    2. 
        
  2. A background worker grinds SHA-256 hashes until it finds one with enough
          leading zero bits — costs you a few seconds of CPU, costs a bot farm a fortune at scale.
    3. 
        
  3. The server checks the proof and issues a short-lived, anonymous session.
    4. 
      


      

        The algorithm & why it works — for the curious
        

          
It's hashcash: a server-set, IP-bound proof-of-work. No CAPTCHA, no third
          party, no image puzzles — just a few seconds of your CPU.


          
1 · Challenge

          
Your browser POSTs /auth/anon/challenge. The server returns a
          stateless, HMAC-signed token (it stores nothing per challenge) carrying a random
          128-bit nonce, a difficulty b in leading-zero bits (~20 by default), a 90-second
          expiry, and an IP tag = HMAC(secret, your IP — the /64 network for
          IPv6).


          
2 · Grind — in a Web Worker, off the UI thread

          
for n = 0, 1, 2, …:
  h = SHA-256(challenge + ":" + n)
  if leadingZeroBits(h) ≥ b:
    solution = n            # found it

          
Finding one takes about 2b hashes on average (geometric
          distribution) — the live console shows the candidate hashes, hashrate and nonce as it
          runs. Verifying it later costs the server exactly one hash.


          
3 · Redeem

          
Your browser POSTs /auth/anon/redeem with {challenge, solution}. The
          server checks, in order: the signature + 90-second freshness + that the IP tag matches
          your network (constant-time), then that SHA-256(challenge + ":" + n) really
          has ≥ b zero bits, then marks the nonce single-use. Only then does it mint
          the session — a JWT with sub = "anon:" + random, empty email,
          anon: true, expiring in {{.AnonTTLHuman}}.


          
Why it works

          
    
            
  • Asymmetric cost. Solving needs ≈ 2b SHA-256 ops; verifying needs
            one. You spend seconds, the server spends microseconds — and a bot farm pays
            that cost per session, forever.
    • 
            
  • Difficulty can't be lowered. b lives inside the signed challenge;
            change it and the HMAC check fails.
    • 
            
  • IP-bound. The puzzle is tied to your network, so the grind can't be farmed
            out to a pool of cheap throwaway IPs — a solution only counts from the address it was
            issued to.
    • 
            
  • One-shot. Each nonce mints exactly one session; replays are rejected.
    • 
            
  • Self-escalating. Repeat hits from one network raise b — every +1 bit
            doubles the work — and both endpoints are rate-limited, so floods get
            exponentially expensive.
    • 
            
  • Domain-separated keys. Challenges are signed with a key derived from
            the master secret and rotated every server boot — distinct from the session-cookie key
            — so a challenge token can never be replayed as a login.
    • 
            
  • Zero identity. The session's subject is random and maps to no record
            anywhere — no email, no profile, no cross-site tracking — and it self-expires.
    •

      Google sign-in — how it works
      
    

    

      
Signing in with Google uses standard OAuth 2.0: Google confirms your email
      address to this site and nothing more — no password is ever shared here.

      
    
        
  1. You authenticate on Google's own page.
    2. 
        
  2. This site receives only your verified email and checks it against the allow-list.
    3. 
        
  3. A signed session cookie is set, scoped to kunaldawn.com and its subdomains.
    4. 
      

      
Best for a device you own and trust: the session is persistent — it lasts
      {{.SessionTTLHuman}}, so you won't have to sign in again for a long while.
